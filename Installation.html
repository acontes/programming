ProActive Installation

ProActive is made available for download to both non-profit organizations (such as universities and research institutions) and commercial companies for evaluation purpose. Please note that the base license does not cover the commercial use of the product for which any interested person or organization is asked to contact the authors : proactive-support@sophia.inria.fr.

Installing ProActive is easy, provided the following steps are followed:

Step 1: Check system requirements

ProActive requires Java 2 (commonly known as JDK 1.3) to be installed on your computer. If you do not yet have it, you can download the JDK 1.3 bundle from Sun's web site.

Please note that ProActive will NOT run with any version prior 1.2 since some features introduced in JDK 1.2 are essential to ProActive. ProActive \emph{should} work with any version of Java to come after JDK 1.3. To check which version of Java your computer is running, simply type the following command line at your command prompt: 

> java -version
java version "1.3.0"
Java(TM) 2 Runtime Environment, Standard Edition (build 1.3.0)
Java HotSpot(TM) Client VM (build 1.3.0, mixed mode)

The output may vary with the operating system or the implementation of Java you're running, but it should always mention 1.3 somewhere, like in the example above.

As ProActive is implemented using Java RMI, you also need to check that your machine supports the TCP/IP network protocol, even if you do not plan to run distributed applications. TCP/IP is always installed by default on Unix machines but may not be installed at all on some Windows machines.

Step 2: Download the archive

You can download the archive file (a standard zip file) containing ProActive from the download section of the ProActive home page. You will be asked to accept the licence agreement and provide a few personal details including your email address. You will then within a few minutes receive an email explaining how to download the archive file.

Step 3: Expand the archive

ProActive comes as a ZIP archive file, which should make it easily expandable under Windows or Unix or any other operating system. Unzip the archive using your favorite ZIP program, such as Winzip under Windows or the unzip command-line utility on most Unix systems. Unzipping the archive creates a ProActive directory and all the files contained in the archive go into this directory and its subdirectories.

Here is a quick overview of the directory structure of the archive:

Directory or File Description
ProActive.jar ProActive bytecode that you need to include in the CLASSPATH in order to use ProActive
ProActive_examples.jar The bytecode and resources of all examples included with ProActive. This jar file needs to be included in the CLASSPATH only when trying to run the examples. All examples rely on ProActive and therefore the ProActive.jar file must be included in the CLASSPATH as well. This is done automatically by the scripts driving the examples. The source code is also included in the src directory (see below)
ic2d.jar The bytecode and resources of IC2D. This jar file needs to be included in the CLASSPATH only when trying to run the application IC2D. IC2D relies on ProActive and therefore the ProActive.jar file must be included in the CLASSPATH. This is done automatically by the scripts launching the application. The source code is also included in the src directory (see below)
src Root directory for the source (.java) files of all examples included with ProActive and of the application IC2D
doc ProActive documentation including the full api doc
scripts/unix Unix sh scripts for running the examples
scripts/windows Windows .bat batch files for running the examples

Step 4 (optional): Update your CLASSPATH

If you use the scripts provided with the distribution to run the examples you do not need to update your classpath

When running java code using ProActive, you need to make sure that the CLASSPATH includes two new entries. The first entry is the JAR file containing ProActive. The second entry is the directory where ProActive outputs the stub classes it generates on-the-fly.

A small utility is provided to output the value of those two entries. It is called install and can be found in the scripts directory located in ProActive. It comes in both flavors: install.sh for Unix systems and install.bat for Windows. 

On a Unix system, the output looks as follows:

bash-2.05$ ./install.sh

----------------------------------------------------------

JVM version: 1.3.0

The two entries you need to add to your CLASSPATH are:

/usr/home/bob/ProActive/ProActive.jar
/usr/home/bob/proactive-tmp


----------------------------------------------------------

On a Windows system, the output looks as follows: 

C:\bob\ProActive\scripts>install.bat

----------------------------------------------------------

JVM version: 1.3.1-rc1

The two entries you need to add to your CLASSPATH are:

C:\bob\ProActive\ProActive.jar
C:\WINNT\Profiles\bob\proactive-tmp


----------------------------------------------------------

Press a key to continue . . .

Of course, the output will look something different on your system. In the example above, the user's home directory is /usr/home/bob and the archive containing ProActive was unzipped directly inside. Note that the directory /usr/home/bob/proactive-tmp was automatically created in order to hold the stub files generated by ProActive.

You do not need to modify your CLASSPATH permanently as long as you include the two entries above using a Java IDE or a shell script. The examples bundled with ProActive set-up the CLASSPATH automatically in order to include those two entries as well as the JAR file containing the examples code. As long as you run the examples from the directory they are located you do not need to change anything.

If not you need to update the environment file that specifies where is the ProActive install directory and the proactive-tmp directory. On Unix systems, the file is called \texttt{env.sh} and is located in ProActive/scripts/unix. On Windows systems, the file is called env.bat and is located in ProActive/scripts/windows.

On Unix systems the file ProActive/scripts/unix/env.sh contains the following values by default : 

    PROACTIVE=../../.
    PROACTIVE_TEMP=$HOME/proactive-tmp

On Windows systems the file ProActive/scripts/windows/env.bat contains the following values by default : 

    set PROACTIVE=..\..\.
    set PROACTIVE_TEMP=%USERPROFILE%\proactive-tmp

By default, the proactive-tmp directory is located inside your home directory. Note that on Windows NT, the home directory is C:\WINNT\Profiles\yourWindowsLoginName and that on Windows 2000 and Windows XP, the home directory is C:\Documents and Settings\yourWindowsLoginName

Step 5 (optional): Create a java.policy file to set permissions

If you use the scripts provided with the distribution to run the examples an existing policy file will be used by default

See Permissions in the JavaTM 2 SDK to learn more about Java permissions. As a first approximation to run your code you can create a simple policy file granted all for everything : 

grant {
    permission java.security.AllPermission;
};

Run a few examples for testing

You are now ready to run the test applications provided with ProActive. Each example comes with a script for launching the application. Depending on you operating system, the script you need to launch is located either in ProActive/scripts/unix ProActive/scripts/windows. The source code of all examples can be found in the directory ProActive/src/org/objectweb/proactive/examples.

Local Example 1: Hello world !

A simple example.

  • script : hello.sh or hello.bat
  • source : ProActive/src/org/objectweb/proactive/examples/hello

Local Example 2: Reader/Writer

This example is the ProActive version of the Readers/Writers canonical problem. To illustrate the ease-of-use of the ProActive model, different synchronization policies can be applied without even stopping the application. This example is based on a easy to use Swing GUI.

  • script : readers.sh or readers.bat
  • source : ProActive/src/org/objectweb/proactive/examples/readers

Local Example 3: The Dining Philosophers

This example is one possible implementation of the well-known Dining Philosophers synchronization problem. This example is based on a easy to use Swing GUI.

  • script : philosophers.sh or philosophers.bat
  • source : ProActive/src/org/objectweb/proactive/examples/philosophers

Troubleshooting and support

If you encounter any problem with installing ProActive and running the examples, please make sure you correctly followed all the steps described above. If it doesn't help, here is a list of the most common mistakes:
  • Java complains about not being able to find ProActive's classes. Your CLASSPATH environment variable does not contain the entry for the directory where ProActive's classes are located. The path to this directory is given to you by the install utility, as described in step 3.
  • ProActive complains about not being able to load stub classes. Your CLASSPATH environment variable does not contain the entry for the directory where ProActive's classes are located. The path to this directory is given to you by the install utility, as described in step 3. Your CLASSPATH environment variable does not contain an entry for the directory where ProActive outputs the stubs classes (proactive-tmp directory). The path to this directory is given to you by the install utility, as described in step 3.
  • ProActive complains about not being to bind to the rmiregistry. The RMIRegistry has not been launched.
  • Java complains about denial of access. If you get the following exceptions, you probably didn't change the file java.policy as described in step 4.
